# Break It Down

## Core process

When facing any problem or decision:

1. **Question the constraint**
   - Why must it be this way?
   - What problem does this actually solve?
   - What would happen if we didn't have this?

2. **Identify fundamental truths**
   - What cannot be changed?
   - What is actually required vs nice-to-have?
   - What is the core job-to-be-done?

3. **Rebuild from basics**
   - Start with only what's essential
   - Add complexity only when justified
   - Question each addition

## Questions to always ask

1. Why does this exist?
2. What would break if we removed it?
3. Is this solving the real problem?
4. Could this be 10x simpler?
5. What are we optimising for?

## Red flags to challenge

- "That's how it's always done"
- "Other platforms do it this way"
- "It's industry standard"
- "We need this for completeness"
- Complex solutions to simple problems

## Application

### Feature design

Instead of: "We need to support every use case"
Ask: "What's the core job-to-be-done?"
Then: Go deep on what matters before expanding breadth

### Technical architecture

Instead of: "We need microservices"
Ask: "What problem are we solving?"
Then: Maybe good module boundaries suffice

### Product decisions

Instead of: "We need a full self-service platform"
Ask: "What actually gets value to users?"
Then: Maybe a simpler path exists before building the full thing
